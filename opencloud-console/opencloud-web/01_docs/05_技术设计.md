### 2.1 Spring Boot 启动类设计

```java
@SpringBootApplication
@MapperScan("com.qoobot.opencloud.**.mapper")
@EnableAsync
@EnableScheduling
@EnableTransactionManagement
public class OpenCloudApplication {

    public static void main(String[] args) {
        SpringApplication app = new SpringApplication(OpenCloudApplication.class);
        // 自定义 Banner
        app.setBannerMode(Banner.Mode.CONSOLE);
        app.run(args);
    }

    /**
     * 应用启动完成回调：打印安全警告
     */
    @EventListener(ApplicationReadyEvent.class)
    public void onApplicationReady(ApplicationReadyEvent event) {
        Environment env = event.getApplicationContext().getEnvironment();
        String jwtSecret = env.getProperty("opencloud.jwt.secret", "");
        if (jwtSecret.contains("opencloud-jwt-secret")) {
            log.warn("⚠️  [SECURITY] JWT_SECRET 使用了默认值，生产环境必须通过环境变量覆盖！");
        }
        String encryptKey = env.getProperty("opencloud.encrypt.key", "");
        if (encryptKey.contains("opencloud-aes-key")) {
            log.warn("⚠️  [SECURITY] ENCRYPT_KEY 使用了默认值，生产环境必须通过环境变量覆盖！");
        }
    }
}
```

### 2.2 多 Profile 配置加载机制

Spring Boot 配置文件加载优先级（从高到低）：

```
1. 命令行参数（--server.port=9090）
2. 环境变量（SPRING_DATASOURCE_URL）
3. application-{profile}.yml（通过 spring.profiles.active 激活）
4. application.yml（基础配置）
5. @PropertySource 注解指定文件
```

**设计原则：**
- `application.yml`：存放所有配置项的默认值（适合开发/本地运行）
- `application-dev.yml`：覆盖开发专属配置（开启 SQL 打印、关闭验证码）
- `application-prod.yml`：覆盖生产专属配置（关闭 Swagger、增强安全）
- 所有敏感参数通过 `${ENV_VAR:defaultValue}` 语法从环境变量注入

### 2.3 Flyway 迁移设计

**配置策略：**

```yaml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration        # 迁移脚本目录
    baseline-on-migrate: true                # 首次部署时基线化
    baseline-version: 0                      # 基线版本号
    validate-on-migrate: true                # 执行前校验 checksum
    out-of-order: false                      # 禁止乱序执行
    clean-disabled: true                     # 禁止 flyway clean（防止意外清库）
    placeholder-replacement: false           # 禁用占位符替换（防止 ${} 被误解析）
```

**脚本命名规范：**
```
V{major}.{minor}.{patch}__{description}.sql
示例：
  V1.0.0__init.sql
  V1.1.0__add_cloud_tables.sql
  V1.2.0__add_monitor_tables.sql
  V1.2.1__add_silence_index.sql
```

**Flyway 执行流程：**
```
应用启动
  │
  ├─ 检查 flyway_schema_history 表是否存在
  │   → 不存在：创建表，记录 baseline 版本（V0）
  │
  ├─ 扫描 classpath:db/migration/ 下所有 V*.sql 文件
  │
  ├─ 计算每个脚本的 checksum（CRC32）
  │   → 与 flyway_schema_history 中已记录的 checksum 比对
  │   → 不匹配：抛出 FlywayException，阻止启动
  │
  ├─ 找出所有未执行的新版本脚本（按版本号升序排列）
  │   → 在同一事务内逐一执行
  │   → 执行成功：写入 flyway_schema_history（version, checksum, executed_at）
  │   → 执行失败：事务回滚，抛出异常，阻止启动
  │
  └─ 所有脚本执行完毕，应用继续启动
```

### 2.4 CORS 跨域处理机制

**实现方案：** 使用 `WebMvcConfigurer.addCorsMappings()` 全局配置（优先级高于 `@CrossOrigin` 注解），通过 Spring Security 的 `cors()` 委托给 MVC 层处理。

```java
// SecurityConfig 中启用 CORS 支持
http.cors(cors -> cors.configurationSource(corsConfigurationSource()));

@Bean
CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOriginPatterns(Arrays.asList(allowedOrigins.split(",")));
    config.setAllowedMethods(List.of("GET","POST","PUT","DELETE","PATCH","OPTIONS"));
    config.setAllowedHeaders(List.of("Content-Type","Authorization","X-User-Id","X-Old-Jti"));
    config.setExposedHeaders(List.of("Authorization"));
    config.setAllowCredentials(false);   // JWT 不依赖 Cookie
    config.setMaxAge(3600L);
    UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/api/**", config);
    return source;
}
```

**关键设计决策：**
- `allowCredentials = false`：JWT 无状态，不依赖 Cookie，不需要携带凭证
- 不使用 `*` 通配来源 + `allowCredentials = true`（浏览器安全策略不允许）
- `allowedOriginPatterns` 支持通配（如 `http://localhost:*` 匹配所有本地端口）

### 2.5 请求链路追踪（traceId）

**实现方案：**

```java
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class TraceIdFilter implements Filter {

    private static final String TRACE_ID_KEY = "traceId";
    private static final String TRACE_ID_HEADER = "X-Trace-Id";

    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
            throws IOException, ServletException {
        HttpServletRequest request = (HttpServletRequest) req;
        // 优先从 Header 中读取（支持网关传递）
        String traceId = request.getHeader(TRACE_ID_HEADER);
        if (!StringUtils.hasText(traceId)) {
            traceId = UUID.randomUUID().toString().replace("-", "").substring(0, 16);
        }
        MDC.put(TRACE_ID_KEY, traceId);
        // 响应头也携带 traceId，便于前端联调
        ((HttpServletResponse) res).setHeader(TRACE_ID_HEADER, traceId);
        try {
            chain.doFilter(req, res);
        } finally {
            MDC.clear();  // 请求结束后清理，防止内存泄漏
        }
    }
}
```

**日志格式中的 traceId：**
```
logback-spring.xml 中通过 %X{traceId} 引用 MDC 中的 traceId：
```

### 2.6 全局异常处理设计

**异常层级体系：**

```
Throwable
  └── Exception
        ├── RuntimeException
        │     ├── BusinessException        # 业务异常基类（所有子模块异常继承）
        │     │     ├── AuthException      # 认证授权异常（auth 模块）
        │     │     └── CloudException     # 云平台异常（cloud 模块）
        │     └── SystemException          # 系统级异常（500）
        └── 其他 Checked Exception
```

**GlobalExceptionHandler 处理优先级：**

| 优先级 | 异常类型 | HTTP 状态 | 响应码 |
|-------|---------|:---:|-------|
| 1 | `MethodArgumentNotValidException` | 400 | 字段错误列表 |
| 2 | `ConstraintViolationException` | 400 | 字段错误列表 |
| 3 | `HttpMessageNotReadableException` | 400 | `SYS_0002` |
| 4 | `MissingServletRequestParameterException` | 400 | `SYS_0003` |
| 5 | `AuthException`（含子类） | 对应状态 | `AUTH_XXXX` |
| 6 | `BusinessException`（其他子类） | 400/422 | `BIZ_XXXX` |
| 7 | `AccessDeniedException` | 403 | `AUTH_2001` |
| 8 | `AuthenticationException` | 401 | `AUTH_1001` |
| 9 | `NoHandlerFoundException` | 404 | `SYS_0404` |
| 10 | `Exception`（兜底） | 500 | `SYS_0001` |

### 2.7 异步线程池设计

**线程池参数决策依据：**

```
场景分析：
- opencloud-monitor：告警 Webhook 写入 + 通知分发（高频 @Async）
- opencloud-system：操作日志写入（每次请求触发，高频 @Async）
- opencloud-cloud：定时同步任务（低频，但可能耗时）

预估并发：
- 告警高峰期：50~100 个告警同时触发
- 日常请求：10~30 QPS
- 通知分发：每次告警触发 1~5 个通知发送

线程池参数：
- corePoolSize = 10：常态并发覆盖
- maxPoolSize = 50：告警高峰扩展
- queueCapacity = 100：防止突发堆积
- 拒绝策略：CallerRunsPolicy（调用方线程执行，降级但不丢失）
```

### 2.8 定时任务设计

**调度线程池设计：**

```
云平台定时同步任务（opencloud-cloud）：
  - @Scheduled(fixedDelay = 3600000)  集群状态同步（每小时）
  - @Scheduled(cron = "0 */5 * * * *") 资源使用量同步（每 5 分钟）

监控静默规则过期清理：
  - @Scheduled(cron = "0 0 2 * * *")  每天凌晨 2 点清理过期静默规则

使用多线程调度池（poolSize=10），避免单个耗时任务阻塞其他调度任务。
```

#### 2.9.2 逻辑删除

```yaml
mybatis-plus:
  global-config:
    db-config:
      logic-delete-field: deleted       # 字段名
      logic-delete-value: 1             # 已删除标志
      logic-not-delete-value: 0         # 未删除标志
```

- Entity 中 `deleted` 字段加 `@TableLogic` 注解（可选，全局配置已生效）
- `BaseMapper.deleteById()` 自动转为 `UPDATE ... SET deleted=1`
- `BaseMapper.selectById()` 自动拼接 `WHERE deleted=0`

#### 2.9.3 自动填充

通过 `MetaObjectHandler` 实现 `created_at`、`updated_at`、`created_by`、`updated_by` 自动填充：

```java
@Component
public class OpenCloudMetaObjectHandler implements MetaObjectHandler {

    @Override
    public void insertFill(MetaObject metaObject) {
        this.strictInsertFill(metaObject, "createdAt", LocalDateTime.class, LocalDateTime.now());
        this.strictInsertFill(metaObject, "updatedAt", LocalDateTime.class, LocalDateTime.now());
        Long userId = getCurrentUserId();
        if (userId != null) {
            this.strictInsertFill(metaObject, "createdBy", Long.class, userId);
            this.strictInsertFill(metaObject, "updatedBy", Long.class, userId);
        }
    }

    @Override
    public void updateFill(MetaObject metaObject) {
        this.strictUpdateFill(metaObject, "updatedAt", LocalDateTime.class, LocalDateTime.now());
        Long userId = getCurrentUserId();
        if (userId != null) {
            this.strictUpdateFill(metaObject, "updatedBy", Long.class, userId);
        }
    }

    private Long getCurrentUserId() {
        try {
            return SecurityUtils.getCurrentUserId();
        } catch (Exception e) {
            return null;  // 未登录场景（如 Flyway 执行时）不强制要求
        }
    }
}
```

### 2.10 优雅停机实现

```yaml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

**附加配置：**

```java
// AsyncConfig 中配置优雅停机
executor.setWaitForTasksToCompleteOnShutdown(true);
executor.setAwaitTerminationSeconds(30);

// SchedulerConfig 中配置优雅停机
scheduler.setWaitForTasksToCompleteOnShutdown(true);
scheduler.setAwaitTerminationSeconds(30);
```

**停机信号处理顺序：**
1. Spring 容器接收 SIGTERM
2. `server.shutdown: graceful` → Tomcat 停止接受新请求（返回 503）
3. 等待正在处理的 HTTP 请求完成（最长 30 秒）
4. 触发 `@PreDestroy` / `DisposableBean.destroy()`
5. 线程池优雅关闭（等待队列任务完成，最长 30 秒）
6. Spring 容器销毁所有 Bean
7. JVM 退出

## 3. 安全设计

### 3.1 密钥管理

| 密钥 | 存储方式 | 强度要求 | 轮换策略 |
|------|---------|---------|---------|
| JWT 签名密钥（`JWT_SECRET`） | 环境变量 | ≥ 256 bits（32 字符） | 密钥轮换时需触发全局强制下线（tokenVersion +1） |
| AES-256-GCM 加密密钥（`ENCRYPT_KEY`） | 环境变量 | 32 字节 | 轮换时需重新加密数据库中所有加密字段 |
| 数据库密码（`DB_PASS`） | 环境变量 | ≥ 16 字符 | 按安全规范定期轮换 |
| Redis 密码（`REDIS_PASS`） | 环境变量 | ≥ 16 字符 | 按安全规范定期轮换 |

### 3.2 生产安全 Checklist

| 检查项 | 状态判断 |
|-------|---------|
| ☐ `JWT_SECRET` 使用随机强密钥（≥32 位），非默认值 | 启动时 warn 检测 |
| ☐ `ENCRYPT_KEY` 使用随机强密钥（32 字节），非默认值 | 启动时 warn 检测 |
| ☐ `knife4j.enable=false` 关闭 Swagger | prod profile 已配置 |
| ☐ `ALLOWED_ORIGINS` 明确配置前端域名 | 需手动配置 |
| ☐ `/actuator/prometheus` 访问 IP 白名单 | Nginx ACL 配置 |
| ☐ 数据库账号最小权限原则（只赋 CRUD，不赋 DDL） | 数据库初始化时手动设置 |
| ☐ Redis 配置密码认证 | 环境变量 `REDIS_PASS` |
| ☐ 开启 HTTPS（Nginx 反代） | 运维层配置 |
| ☐ 请求体日志中密码字段已脱敏 | `RequestLoggingFilter` 已实现 |

## 4. 性能设计

### 4.1 数据库连接池调优

| 参数 | 开发值 | 生产值 | 说明 |
|------|-------|-------|------|
| `minimum-idle` | 5 | 10 | 最小空闲连接数 |
| `maximum-pool-size` | 20 | 50 | 最大连接数 |
| `connection-timeout` | 30000ms | 30000ms | 获取连接超时 |
| `idle-timeout` | 600000ms | 600000ms | 空闲连接回收（10min） |
| `connection-test-query` | `SELECT 1` | `SELECT 1` | 连接保活验证 |
| `max-lifetime` | 1800000ms | 1800000ms | 连接最大存活（30min） |

**连接池容量估算：**
```
预估 TPS：100
平均查询耗时：20ms
所需连接数 = TPS * 平均耗时 / 1000 = 100 * 20 / 1000 = 2
峰值预留 10x 安全系数 → max-pool-size = 20（开发） / 50（生产）
```

```java
@Component
public class CloudPlatformHealthIndicator implements HealthIndicator {

    @Override
    public Health health() {
        // 检查是否有至少一个启用的集群配置
        long enabledClusters = clusterConfigService.countEnabled();
        if (enabledClusters > 0) {

                    .withDetail("enabledClusters", enabledClusters)
```

## 8. 高可用设计

### 8.1 当前单节点架构

```
[Nginx/LB]
     │
     ▼
[opencloud-console:8080]  ← 单节点
     │
     ├─ PostgreSQL（主从复制）
     └─ Redis（Standalone / Sentinel）
```

### 8.2 水平扩展注意事项

当部署多个实例时，以下功能需要特殊处理：

| 功能 | 问题 | 方案 |
|------|------|------|
| **WebSocket 告警推送** | 不同节点的 WebSocket 连接无法互通 | 迁移至 Redis Pub/Sub：publish 到频道，所有节点订阅后广播给本节点连接的客户端 |
| **定时任务** | 多节点同时执行定时任务，导致重复执行 | 引入分布式锁（Redis SETNX）或使用 ShedLock 框架 |
| **本地缓存** | Caffeine 等本地缓存节点间不一致 | 不使用本地缓存，统一使用 Redis |

> **当前版本**（v1.0）仅支持单节点部署；水平扩展能力作为 v2.0 迭代目标。
