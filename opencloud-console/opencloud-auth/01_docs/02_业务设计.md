# opencloud-auth 业务设计文档

---

## 1. 业务背景

opencloud-auth 模块承载 OpenCloud 云管平台的统一认证与授权能力，是整个平台安全体系的核心入口。

### 1.1 使用角色

| 角色 | 说明 |
|------|------|
| 超级管理员 | 平台全局权限，管理租户、角色、用户，不受 RBAC 限制 |
| 租户管理员 | 管理所属租户下的用户和角色，权限边界为本租户 |
| 普通用户 | 仅拥有被授权的资源操作权限，受 RBAC 严格约束 |

### 1.2 核心职责

| 职责 | 说明 |
|------|------|
| 身份认证（Authentication） | 验证用户是谁，颁发 JWT Token |
| 授权鉴权（Authorization） | 验证用户能做什么，基于 RBAC 模型校验权限 |
| Token 生命周期管理 | 颁发、刷新、注销、黑名单管理 |
| 安全防护 | 登录限频、账号锁定、验证码、重放攻击防护 |
| 操作审计 | 记录登录、登出、密码修改等关键操作日志 |

---

## 2. 业务目标

1. 提供统一登录入口，支持账号密码登录
2. 支持多租户隔离，租户间用户、角色、权限互不干扰
3. 基于 RBAC 模型管理权限，精确到菜单、按钮、API 接口
4. 提供 Token 颁发、校验、刷新、注销的完整生命周期管理
5. 登录行为可追溯，记录登录成功、登录失败、登出等操作日志

---

## 3. 核心业务流程

### 3.1 用户登录流程

**触发条件：** 用户在登录页提交账号、密码（以及可选的验证码）。

**前置条件：** 登录接口已开放白名单（不校验 Token）。

```
用户提交账号 + 密码（+ 可选 captchaKey + captchaCode）
  │
  ├─[启用验证码]→ 从 Redis 取出 captchaKey 对应的验证码文本
  │               → 比对验证码（不区分大小写）
  │               → 比对通过后立即从 Redis 删除（一次性）
  │               → 比对失败 → 返回 CAPTCHA_ERROR
  │
  ├─ 查询 auth_user_credential（by username）
  │   → 账号不存在 → 返回通用错误（不暴露账号/密码哪个错）
  │
  ├─ 检查账号状态
  │   → status = DISABLED → 返回 ACCOUNT_DISABLED
  │   → status = LOCKED & 锁定未到期 → 返回 ACCOUNT_LOCKED + 剩余秒数
  │   → status = LOCKED & 锁定已到期 → 自动解锁，继续流程
  │
  ├─ BCrypt 比对密码
  │   → 失败 → login_fail_count +1
  │           → fail_count >= 5 → 更新状态为 LOCKED，设置 lock_expire_time = now + 15min
  │           → 返回通用错误 + 剩余可尝试次数提示
  │   → 成功 → 重置 login_fail_count = 0，last_login_time = now
  │
  ├─ BCrypt 比对失败时：写入登录失败日志
  │   字段: userId, tenantId, loginIp, loginTime, userAgent, status=FAILED, failReason=PASSWORD_ERROR
  │
  ├─ 读取 Redis auth:token:version:{userId}（不存在则初始化为 0）
  │
  ├─ 生成 AccessToken（JWT）
  │   Claims: userId, tenantId, username, roles[], jti（UUID）, tokenVersion
  │   有效期: 2 小时
  │
  ├─ 生成 RefreshToken（UUID）
  │   → 存入 Redis：key = auth:refresh:{userId}:{jti}, value = RefreshToken 详情
  │   TTL: 7 天
  │
  ├─ 写入登录日志（login_log）
  │   字段: userId, tenantId, loginIp, loginTime, userAgent, status=SUCCESS
  │
  └─ 返回响应
      {
        accessToken, accessTokenExpireAt,
        refreshToken, refreshTokenExpireAt,
        userId, username, nickname, tenantId
      }
```

**后置条件：** 客户端持有有效的 AccessToken + RefreshToken，可访问受保护资源。

### 3.2 Token 刷新流程

**触发条件：** 客户端检测到 AccessToken 过期（HTTP 401），携带 RefreshToken 请求刷新。

**前置条件：** RefreshToken 接口已开放白名单（不校验 AccessToken）。

```
客户端携带 RefreshToken 请求 /auth/token/refresh
  │
  ├─ 解析 RefreshToken（JWT 或查 Redis）
  │   → 格式非法 / 找不到 → 返回 401 REFRESH_TOKEN_INVALID
  │
  ├─ 检查 Redis 中 key = auth:refresh:{userId}:{jti} 是否存在
  │   → 不存在（已过期 or 已被使用）→ 返回 401 REFRESH_TOKEN_EXPIRED_OR_USED
  │
  ├─ 比对 Redis 存储值与请求中 RefreshToken 是否一致
  │   → 不一致（疑似重放攻击）→ 删除该用户所有 RefreshToken，返回 401 REPLAY_ATTACK
  │
  ├─ 立即删除旧 RefreshToken（del auth:refresh:{userId}:{jti}）
  │
  ├─ 读取 Redis auth:token:version:{userId} 当前版本号
  │
  ├─ 生成新 AccessToken（新 jti，携带当前 tokenVersion）
  ├─ 生成新 RefreshToken
  │   → 写入 Redis：key = auth:refresh:{userId}:{newJti}，TTL 7 天
  │
  └─ 返回新 accessToken + 新 refreshToken
```

### 3.3 用户登出流程

**触发条件：** 用户主动点击退出登录，或客户端检测到用户会话应终止。

```
客户端携带 AccessToken 请求 /auth/logout
  │
  ├─ JwtAuthenticationFilter 校验 Token 有效性（已是通用流程，此处省略细节）
  │
  ├─ 从 Token Claims 中提取 jti、userId、expireAt
  │
  ├─ 将 jti 写入 Token 黑名单
  │   key = auth:blacklist:{jti}，value = "1"
  │   TTL = Token 剩余有效期（避免 Redis 无限膨胀）
  │
  ├─ 删除 Redis 中该 Token 关联的 RefreshToken
  │   key = auth:refresh:{userId}:{jti}
  │
  ├─ 写入登出日志（login_log）
  │   字段: userId, tenantId, logoutTime, status=LOGOUT
  │
  └─ 返回 200 OK
```

### 3.4 权限鉴权流程

**触发条件：** 任意请求到达服务端，由 `JwtAuthenticationFilter` 自动触发（除白名单接口外）。

```
请求到达服务端
  │
  ├─ 检查请求路径是否在白名单中（/auth/login, /auth/token/refresh, /captcha 等）
  │   → 在白名单 → 直接放行，不做 Token 校验
  │
  ├─ 提取 Authorization Header
  │   → 不存在 / 格式非 "Bearer {token}" → 返回 401 UNAUTHORIZED
  │
  ├─ 校验 JWT 签名
  │   → 签名非法 → 返回 401 TOKEN_INVALID
  │
  ├─ 校验 Token 过期时间
  │   → 已过期 → 返回 401 TOKEN_EXPIRED
  │
  ├─ 校验 Token jti 是否在黑名单中（Redis GET auth:blacklist:{jti}）
  │   → 命中黑名单 → 返回 401 TOKEN_REVOKED
  │
  ├─ 校验 tokenVersion
  │   → 读取 Redis auth:token:version:{userId}
  │   → Token 中 tokenVersion < Redis 中版本号 → 返回 401 TOKEN_VERSION_MISMATCH
  │
  ├─ 解析 Claims：userId, tenantId, username, roles[]
  │   → 构建 Authentication 对象，写入 SecurityContext
  │
  ├─ 加载权限集合
  │   → 先查 Redis 缓存：key = auth:perm:{userId}
  │   → 缓存未命中 → 查询数据库（role → permission 关联表）→ 写入 Redis（TTL 10 分钟）
  │   → 超级管理员（role = SUPER_ADMIN）→ 跳过权限匹配，直接放行
  │
  ├─ Spring Security AccessDecisionManager 匹配请求路径 & HTTP 方法
  │   → 所需权限标识 不在 用户权限集合中 → 返回 403 FORBIDDEN
  │
  └─ 放行，业务 Controller 处理请求
```

### 3.5 验证码获取流程

**触发条件：** 登录页加载 / 用户点击刷新验证码。

```
客户端 GET /captcha
  │
  ├─ 生成 4~6 位随机字母数字字符串（验证码文本）
  │
  ├─ 使用图形验证码库（如 EasyCaptcha）渲染为图片
  │
  ├─ 生成 captchaKey = UUID
  │
  ├─ 写入 Redis：key = auth:captcha:{captchaKey}，value = 验证码文本（统一转小写存储）
  │   TTL: 5 分钟
  │
  └─ 返回响应
      {
        captchaKey,          // 客户端登录时需携带
        captchaImage         // Base64 编码的图片数据（data:image/png;base64,...）
      }
```

### 3.6 修改密码流程

**触发条件：** 已登录用户在个人设置页提交修改密码请求。

**前置条件：** 用户已持有有效 AccessToken。

```
客户端携带 AccessToken 请求 POST /auth/password/change
请求体: { oldPassword, newPassword, confirmPassword }
  │
  ├─ JwtAuthenticationFilter 校验 Token 有效性
  │
  ├─ 校验 newPassword == confirmPassword
  │   → 不一致 → 返回 400 PASSWORD_CONFIRM_MISMATCH
  │
  ├─ 校验 newPassword 强度（≥8位，含大小写字母+数字）
  │   → 不符合 → 返回 400 PASSWORD_TOO_WEAK
  │
  ├─ 校验 newPassword != oldPassword
  │   → 相同 → 返回 400 PASSWORD_SAME_AS_OLD
  │
  ├─ 查询 auth_user_credential（by userId from SecurityContext）
  │
  ├─ BCrypt 比对 oldPassword
  │   → 失败 → 返回 400 OLD_PASSWORD_INCORRECT
  │
  ├─ BCrypt 加密 newPassword，更新 auth_user_credential.password_hash
  │   同时更新 password_update_time = now
  │
  ├─ 清除该用户所有 RefreshToken
  │   → 扫描并删除 Redis key pattern: auth:refresh:{userId}:*
  │
  ├─ 将当前 AccessToken jti 加入黑名单（此次请求本身使用的 Token 也失效）
  │   key = auth:blacklist:{jti}，TTL = Token 剩余有效期
  │
  ├─ 写入操作日志（operation_log）
  │   字段: userId, tenantId, operationType=CHANGE_PASSWORD, operateTime, operateIp
  │
  └─ 返回 200 OK（提示密码修改成功，请重新登录）
```

### 3.7 获取当前用户信息流程

**触发条件：** 前端应用初始化时（如刷新页面后），需要拉取当前用户的完整信息与权限数据。

```
客户端携带 AccessToken 请求 GET /auth/me
  │
  ├─ JwtAuthenticationFilter 校验 Token 有效性（同 3.4，此处省略）
  │
  ├─ 从 SecurityContext 取出 userId、tenantId
  │
  ├─ 查询用户基础信息（联查 system 模块 user 表 或 本地缓存）
  │   字段: userId, username, nickname, avatar, email, tenantId, tenantName
  │
  ├─ 查询用户角色列表
  │   → 先查 Redis 缓存：key = auth:perm:{userId}
  │   → 缓存未命中 → 查 user_role 关联表 → 取角色 code 列表
  │
  ├─ 查询权限标识集合（permission codes）
  │   → 由角色关联的菜单、按钮、API 权限 code 汇总
  │   → 写入 Redis 缓存（TTL 10 分钟）
  │
  └─ 返回响应
      {
        userId, username, nickname, avatar,
        tenantId, tenantName,
        roles: ["ROLE_ADMIN", ...],
        permissions: ["system:user:list", "system:user:add", ...]
      }
```

### 3.8 强制下线流程

**触发条件：** 管理员在用户管理页对指定用户执行"禁用账号"或"强制踢出"操作。

```
管理员请求 POST /auth/admin/force-logout/{targetUserId}
  │
  ├─ 校验操作者权限（需具备 auth:admin:force-logout 权限）
  │
  ├─[禁用账号]
  │   → 更新 auth_user_credential.status = DISABLED
  │
  ├─[踢出用户（不禁用）]
  │   → 不修改账号状态，仅使 Token 失效
  │
  ├─ 清除 Redis 中该用户所有 RefreshToken
  │   → 扫描删除 key pattern: auth:refresh:{targetUserId}:*
  │
  ├─ 将该用户当前有效的 AccessToken jti 加入黑名单
  │   注：JWT 无状态，服务端无法枚举所有已颁发 Token；
  │       实践上通过以下策略覆盖：
  │       - 将 {userId}:{tokenVersion} 版本号存入 Redis
  │       - 颁发 Token 时携带 tokenVersion，鉴权时比对版本
  │       - 强制下线时将 Redis 中 tokenVersion +1，所有旧 Token 自动失效
  │       key = auth:token:version:{userId}，初始值 = 0
  │
  ├─ 写入操作日志
  │   字段: operatorId, targetUserId, operationType=FORCE_LOGOUT, operateTime
  │
  └─ 返回 200 OK
```

> **tokenVersion 机制说明：** 在 Token Claims 中增加 `tokenVersion` 字段，鉴权时从 Redis 中读取最新版本比对。强制下线时 Redis 中版本 +1，所有携带旧版本的 Token 即被视为无效，无需枚举已颁发 Token。

### 3.9 管理员解锁账号流程

**触发条件：** 管理员在用户管理页对被锁定账号执行"手动解锁"操作。

```
管理员请求 POST /auth/admin/unlock/{targetUserId}
  │
  ├─ 校验操作者权限（需具备 auth:admin:unlock 权限）
  │
  ├─ 查询 auth_user_credential（by targetUserId）
  │   → 账号不存在 → 返回 404
  │   → 账号状态非 LOCKED → 返回 400（账号未处于锁定状态）
  │
  ├─ 更新 auth_user_credential
  │   → status = ACTIVE
  │   → login_fail_count = 0
  │   → lock_expire_time = null
  │
  ├─ 写入操作日志
  │   字段: operatorId, targetUserId, operationType=UNLOCK_ACCOUNT, operateTime
  │
  └─ 返回 200 OK
```

## 4. 业务规则

### 4.1 密码规则

| 规则 | 说明 |
|------|------|
| 密码强度 | 至少 8 位，必须同时包含大写字母、小写字母、数字；不允许纯数字或纯字母 |
| 新旧密码不同 | 修改密码时，新密码不得与旧密码相同 |
| 密码存储 | 使用 BCrypt 加密存储，禁止明文或弱哈希（MD5/SHA1） |
| 密码修改后强制下线 | 密码修改成功后，该账号所有在线 Token 立即失效，需重新登录 |

### 4.2 账号安全规则

| 规则 | 说明 |
|------|------|
| 登录失败锁定 | 连续登录失败 5 次，账号锁定 15 分钟；锁定期间拒绝登录请求 |
| 自动解锁 | 锁定期满后，下次登录请求自动解锁，不需要人工干预 |
| 管理员解锁 | 管理员可手动解锁账号（更新状态 & 清零失败次数） |
| 登录限频 | 同一 IP 每分钟登录请求不超过 10 次，超限返回 429；基于 Redis 计数器实现 |

### 4.3 Token 规则

| 规则 | 说明 |
|------|------|
| AccessToken 有效期 | 2 小时，JWT 格式，包含 userId / tenantId / username / roles / jti / tokenVersion |
| RefreshToken 有效期 | 7 天，UUID 格式，存储于 Redis |
| RefreshToken 单次使用 | 每次刷新后旧 RefreshToken 立即废弃，签发全新 RefreshToken（防重放攻击） |
| Token 黑名单 | 登出 / 密码修改 / 强制下线时，AccessToken jti 写入 Redis 黑名单，TTL = Token 剩余有效期 |
| tokenVersion 版本控制 | 强制下线时 Redis 中版本号 +1，所有旧 Token 自动失效（无需枚举颁发记录） |

### 4.4 验证码规则

| 规则 | 说明 |
|------|------|
| 验证码有效期 | 5 分钟，存储于 Redis |
| 验证码一次有效 | 校验通过后立即从 Redis 删除，防止重复使用 |
| 大小写不敏感 | 验证码统一转小写后比对 |
| 是否强制验证码 | 可通过配置项控制，建议生产环境默认开启 |

### 4.5 权限与租户规则

| 规则 | 说明 |
|------|------|
| 租户隔离 | Token 中携带 tenantId，所有数据查询均以 tenantId 作为隔离条件；跨租户访问直接拒绝，返回 403 |
| 超级管理员 | role = SUPER_ADMIN 时，跳过所有权限校验，直接放行 |
| 权限缓存 | 用户权限集合缓存于 Redis（TTL 10 分钟）；角色/权限变更时由 system 模块主动清除缓存 |
| 白名单接口 | /auth/login、/auth/token/refresh、/captcha 等接口不需要 Token，配置在 SecurityConfig 中 |

## 5. 业务异常场景

### 5.1 登录相关

| 场景 | HTTP 状态 | 错误码 | 处理方式 |
|------|-----------|--------|---------|
| 账号不存在 | 200 | AUTH_0001 | 返回通用错误"账号或密码错误"，不透露具体原因 |
| 密码错误 | 200 | AUTH_0001 | 同上，与"账号不存在"返回同一错误，防止账号枚举 |
| 账号被禁用 | 200 | AUTH_0002 | 返回"账号已被禁用，请联系管理员" |
| 账号被锁定 | 200 | AUTH_0003 | 返回"账号已锁定，剩余 X 分钟解锁" |
| 验证码错误 | 200 | AUTH_0004 | 返回验证码错误提示，不计入登录失败次数 |
| 验证码已过期 | 200 | AUTH_0005 | 返回"验证码已过期，请重新获取" |
| 登录请求超频 | 429 | AUTH_0006 | 返回"操作过于频繁，请稍后再试" |

### 5.2 Token 相关

| 场景 | HTTP 状态 | 错误码 | 处理方式 |
|------|-----------|--------|---------|
| Token 不存在 / 格式错误 | 401 | AUTH_1001 | 返回"未授权，请登录" |
| Token 签名非法 | 401 | AUTH_1002 | 返回"Token 无效" |
| Token 已过期 | 401 | AUTH_1003 | 返回"Token 已过期"，客户端使用 RefreshToken 刷新 |
| Token 在黑名单中 | 401 | AUTH_1004 | 返回"Token 已注销，请重新登录" |
| tokenVersion 不匹配 | 401 | AUTH_1005 | 返回"登录状态已失效，请重新登录" |
| RefreshToken 不存在 / 已过期 | 401 | AUTH_1006 | 返回"登录已过期，请重新登录" |
| RefreshToken 疑似重放 | 401 | AUTH_1007 | 清除该用户所有 RefreshToken，返回"安全风险，请重新登录" |

### 5.3 权限相关

| 场景 | HTTP 状态 | 错误码 | 处理方式 |
|------|-----------|--------|---------|
| 无接口访问权限 | 403 | AUTH_2001 | 返回"无权限，请联系管理员" |
| 跨租户访问 | 403 | AUTH_2002 | 返回"禁止跨租户访问" |

### 5.4 密码修改相关

| 场景 | HTTP 状态 | 错误码 | 处理方式 |
|------|-----------|--------|---------|
| 旧密码错误 | 400 | AUTH_3001 | 返回"旧密码不正确" |
| 新密码强度不足 | 400 | AUTH_3002 | 返回"密码须至少 8 位，包含大小写字母和数字" |
| 新旧密码相同 | 400 | AUTH_3003 | 返回"新密码不能与旧密码相同" |
| 两次密码不一致 | 400 | AUTH_3004 | 返回"两次输入密码不一致" |

### 5.5 管理操作相关

| 场景 | HTTP 状态 | 错误码 | 处理方式 |
|------|-----------|--------|---------|
| 解锁目标账号不存在 | 404 | AUTH_4001 | 返回"用户不存在" |
| 解锁目标账号未锁定 | 400 | AUTH_4002 | 返回"账号当前未处于锁定状态" |
| 无管理操作权限 | 403 | AUTH_2001 | 同权限相关，返回"无权限，请联系管理员" |

## 6. 关键数据说明

### 6.1 AccessToken Claims 结构

| 字段 | 类型 | 说明 |
|------|------|------|
| sub | String | 用户 ID |
| tenantId | String | 租户 ID |
| username | String | 登录账号 |
| roles | List\<String\> | 角色 code 列表，如 `["ROLE_ADMIN"]` |
| jti | String | Token 唯一 ID（UUID），用于黑名单校验 |
| tokenVersion | Integer | Token 版本号，与 Redis 中版本比对 |
| iat | Long | 签发时间（Unix 时间戳） |
| exp | Long | 过期时间（Unix 时间戳） |

### 6.2 Redis Key 设计

| Key 模式 | 说明 | TTL |
|----------|------|-----|
| `auth:captcha:{captchaKey}` | 验证码文本 | 5 分钟 |
| `auth:refresh:{userId}:{jti}` | RefreshToken 详情 | 7 天 |
| `auth:blacklist:{jti}` | Token 黑名单标记 | Token 剩余有效期 |
| `auth:perm:{userId}` | 用户权限集合缓存 | 10 分钟 |
| `auth:token:version:{userId}` | Token 版本号（强制下线用） | 永久（账号注销时清除） |
| `auth:login:rate:{ip}` | 登录限频计数器 | 1 分钟 |

### 6.3 登录日志记录内容

| 字段 | 说明 |
|------|------|
| userId | 登录用户 ID |
| tenantId | 所属租户 ID |
| loginIp | 客户端 IP（取 X-Forwarded-For 或 RemoteAddr） |
| userAgent | 浏览器 / 客户端标识 |
| loginTime | 登录时间 |
| logoutTime | 登出时间（登出时补填） |
| status | SUCCESS / FAILED / LOGOUT / FORCE_LOGOUT |
| failReason | 失败原因（账号不存在 / 密码错误 / 账号锁定等） |

## 7. 与其他模块的业务关联

| 关联模块 | 交互方向 | 交互说明 |
|----------|---------|---------|
| opencloud-system | auth 消费 system 数据 | 用户基础信息、角色、菜单、权限由 system 模块维护；auth 通过内部接口或共享数据库查询 |
| opencloud-system | system 通知 auth | 角色/权限/用户状态变更时，system 主动调用 auth 接口清除对应用户的权限缓存（`auth:perm:{userId}`） |
| opencloud-common | auth 依赖 common | 使用公共异常码定义、统一响应体（Result\<T\>）、日志工具、IP 解析工具 |
