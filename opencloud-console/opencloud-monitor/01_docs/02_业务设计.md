# opencloud-monitor 业务设计文档

---

## 1. 业务背景

opencloud-monitor 模块承载 OpenCloud 云管平台的统一监控与告警能力，是平台运维体系的核心入口。

### 1.1 使用角色

| 角色 | 说明 |
|------|------|
| 超级管理员 | 管理全局监控配置（告警规则、通知渠道），可查看所有租户的监控数据 |
| 租户管理员 | 管理本租户的告警规则和通知渠道，仅能查看本租户资源的监控数据 |
| 普通用户 | 只读查看监控仪表盘和告警记录，不可修改规则和渠道配置 |
| 系统（AlertManager） | 通过 Webhook 向本模块推送告警事件，触发通知发送和记录写入 |

### 1.2 核心职责

| 职责 | 说明 |
|------|------|
| 指标查询 | 代理转发 PromQL 查询至 Prometheus，屏蔽后端地址，统一做权限校验 |
| 仪表盘集成 | 集成 Grafana API，提供面板列表和嵌入 URL，前端以 iframe 方式渲染 |
| 告警规则管理 | 在数据库中维护告警规则（指标/阈值/持续时间/严重级别/通知渠道），同步推送至 AlertManager |
| 告警事件处理 | 接收 AlertManager Webhook，写入告警记录，触发 WebSocket 实时推送和多渠道通知 |
| 实时推送 | 通过 WebSocket 将告警事件实时推送至前端在线用户 |
| 通知渠道管理 | 维护邮件/钉钉/企业微信/自定义 Webhook 等通知渠道配置 |
| 告警静默 | 支持在指定时间窗口内屏蔽特定规则的告警通知，避免夜间维护时误报 |

---

## 2. 业务目标

1. 提供统一监控入口，代理 Prometheus 查询，前端无需直连 Prometheus
2. 集成 Grafana，前端通过 iframe 嵌入可视化面板，不需要单独开放 Grafana 外网访问
3. 支持告警规则的可视化管理，规则存储于数据库并同步至 AlertManager
4. 告警发生时实时通知：WebSocket 推送至前端 + 多渠道外部通知（邮件/钉钉/企业微信）
5. 完整的告警生命周期追踪：从触发 → 推送 → 确认 → 解决，全程可查
6. 支持告警静默，减少维护窗口期的噪音告警

## 3. 核心业务流程

### 3.1 Prometheus 指标查询流程

**触发条件：** 前端仪表盘需要渲染图表时，调用监控接口获取时序数据。

**前置条件：** 用户已登录，Token 有效，具备监控查询权限。

```
客户端请求 GET /api/monitor/metrics/query 或 /api/monitor/metrics/query_range
  │
  ├─ JwtAuthenticationFilter 校验 Token 有效性
  │
  ├─ 校验请求参数
  │   → query（PromQL 表达式）必填
  │   → query_range 需额外传 start / end / step
  │   → step 需 > 0，时间范围不超过 30 天
  │
  ├─ 构建 Prometheus API 请求
  │   → 即时查询：GET {prometheus.url}/api/v1/query?query={expr}&time={time}
  │   → 范围查询：GET {prometheus.url}/api/v1/query_range?query={expr}&start={}&end={}&step={}
  │
  ├─ WebClient 发送请求至 Prometheus（带超时：10s）
  │   → Prometheus 不可达 → 返回 503 PROMETHEUS_UNAVAILABLE
  │   → Prometheus 返回 error → 返回 400 PROMETHEUS_QUERY_ERROR（携带 Prometheus 原始错误信息）
  │
  └─ 透传 Prometheus 响应体，返回给前端
```

### 3.2 节点指标查询流程

**触发条件：** 前端节点列表页需要展示各节点的资源使用状态。

```
客户端请求 GET /api/monitor/metrics/nodes
  │
  ├─ JwtAuthenticationFilter 校验 Token 有效性
  │
  ├─ 并行向 Prometheus 查询各节点指标
  │   ├─ CPU：rate(node_cpu_seconds_total{mode!="idle"}[5m]) by (instance)
  │   ├─ 内存：1 - node_memory_MemAvailable_bytes/node_memory_MemTotal_bytes by (instance)
  │   ├─ 磁盘：(node_filesystem_size_bytes - node_filesystem_free_bytes) / node_filesystem_size_bytes by (instance, mountpoint)
  │   └─ 在线状态：up{job="node"} by (instance)
  │
  ├─ 聚合相同 instance 的结果，构建节点指标对象列表
  │
  └─ 返回节点指标列表
      [{
        instance: "192.168.1.10:9100",
        cpuUsage: 0.32,
        memUsage: 0.68,
        diskUsage: 0.45,
        online: true
      }, ...]
```

### 3.3 集群总览指标查询流程

**触发条件：** 前端首页仪表盘加载，需要获取聚合后的集群资源使用概览。

```
客户端请求 GET /api/monitor/metrics/overview
  │
  ├─ 服务端并行执行以下预置 PromQL
  │   ├─ CPU 使用率：1 - avg(rate(node_cpu_seconds_total{mode="idle"}[5m]))
  │   ├─ 内存使用率：1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes
  │   ├─ 磁盘使用率：(node_filesystem_size_bytes - node_filesystem_free_bytes) / node_filesystem_size_bytes
  │   └─ 网络吞吐：sum(rate(node_network_receive_bytes_total[5m])) + sum(rate(node_network_transmit_bytes_total[5m]))
  │
  ├─ 对各指标结果聚合（avg / sum），计算百分比
  │
  └─ 返回结构化总览数据
      {
        cpuUsage: 0.45,        // 45%
        memUsage: 0.72,        // 72%
        diskUsage: 0.38,       // 38%
        networkThroughput: ..., // bytes/s
        timestamp: ...
      }
```

```

  ├─ 调用 Prometheus GET /api/v1/label/__name__/values

  │
  └─ 返回指标名称列表（数组）
      ["node_cpu_seconds_total", "node_memory_MemAvailable_bytes", ...]
```

### 3.5 Grafana 面板集成流程

**触发条件：** 前端需要展示 Grafana 可视化面板时。

> **v1 范围说明：** 仅支持面板列表和嵌入 URL 生成；Grafana Render 截图依赖独立渲染插件，v1 不实现。

```
客户端请求 GET /api/monitor/grafana/dashboards
  │
  ├─ 服务端调用 Grafana API
  │   GET {grafana.url}/api/search?type=dash-db
  │   Header: Authorization: Bearer {grafana.token}
  │
  ├─ Grafana 不可达 → 返回 503 GRAFANA_UNAVAILABLE
  │
  └─ 返回面板列表（uid, title, url, tags, folderId）

客户端请求 GET /api/monitor/grafana/dashboards/{uid}/embed
  │
  ├─ 构建嵌入 URL
  │   {grafana.url}/d/{uid}?orgId=1&kiosk&theme=light&from=now-1h&to=now
  │   （kiosk 模式隐藏 Grafana 导航栏）
  │
  ├─ 若 Grafana 开启了鉴权，在 URL 中追加认证参数
  │
  └─ 返回 { embedUrl: "..." }
```

### 3.6 告警规则管理流程

#### 3.6.1 新增告警规则

**触发条件：** 管理员在告警规则配置页面创建新的告警规则。

```
管理员请求 POST /api/monitor/alerts/rules
  │
  ├─ 校验权限（需具备 monitor:alert:rule:add）
  │
  ├─ 校验请求参数
  │   → ruleName 必填，同租户下唯一
  │   → promqlExpr 必填，格式校验（基本语法检查）
  │   → severity 必填：INFO / WARNING / CRITICAL
  │   → duration 必填（持续时间，如 "5m"），避免毛刺触发
  │   → notifyChannelIds 通知渠道 ID 列表，可为空（仅记录不通知）
  │
  ├─ 试执行 PromQL（调 Prometheus /api/v1/query 确认语法有效）
  │   → 执行失败 → 返回 400 INVALID_PROMQL_EXPR
  │
  ├─ 写入数据库 monitor_alert_rule
  │   → 默认 status = ENABLED
  │
  ├─ 同步至 AlertManager（调用 AlertManager Reload API）
  │   → 调用失败仅记日志，不影响主流程（规则已入库，稍后可重试）
  │
  └─ 返回创建成功的规则详情
```

#### 3.6.2 查询告警规则

```
客户端请求 GET /api/monitor/alerts/rules（分页）
  │
  ├─ 校验权限（需具备 monitor:alert:rule:list）
  ├─ 支持查询参数：ruleName（模糊）/ severity / status / page / size
  ├─ 按 tenantId 过滤（超管可查全部）
  └─ 返回分页列表

客户端请求 GET /api/monitor/alerts/rules/{id}（详情）
  │
  ├─ 校验规则存在
  └─ 返回规则详情（含 notifyChannelIds）
```

#### 3.6.3 编辑告警规则

```
管理员请求 PUT /api/monitor/alerts/rules/{id}
  │
  ├─ 校验权限（需具备 monitor:alert:rule:edit）
  ├─ 校验规则存在
  ├─ 校验修改后 PromQL 语法有效（同新增流程）
  ├─ 更新数据库
  ├─ 同步至 AlertManager
  └─ 返回更新后规则详情
```

#### 3.6.4 删除告警规则

```
管理员请求 DELETE /api/monitor/alerts/rules/{id}
  │
  ├─ 校验权限（需具备 monitor:alert:rule:delete）
  ├─ 逻辑删除（deleted = 1）
  │   → 关联告警记录保留，不级联删除
  ├─ 同步至 AlertManager（移除对应规则）
  └─ 返回 200 OK
```

#### 3.6.5 启用/禁用告警规则

```
管理员请求 PUT /api/monitor/alerts/rules/{id}/status
  │
  ├─ 切换 status = ENABLED / DISABLED
  ├─ 同步至 AlertManager（禁用时移除规则，启用时重新添加）
  └─ 返回 200 OK
```

### 3.7 AlertManager Webhook 接收流程

**触发条件：** AlertManager 检测到告警触发或解除，主动推送至本模块 Webhook 接口。

**前置条件：** AlertManager 配置了本模块 Webhook URL；接口对外开放（在 Spring Security 白名单中）。

```
AlertManager 推送 POST /api/monitor/alerts/webhook
请求体：标准 AlertManager Webhook payload（包含 alerts 数组）
  │
  ├─ 验证签名（若配置了 webhookSecret）
  │   → Header X-Prometheus-Token 不匹配 → 返回 401，记录安全告警日志
  │
  ├─ 遍历 payload.alerts 数组
  │   每条 alert 包含：
  │   - labels（alertname, severity, instance, job 等）
  │   - annotations（summary, description）
  │   - startsAt（告警触发时间）
  │   - endsAt（告警解除时间，firing 时为零值）
  │   - status（firing / resolved）
  │
  ├─[status = firing]
  │   ├─ 根据 labels.alertname 匹配数据库中的告警规则（monitor_alert_rule）
  │   │   → 未匹配到规则 → 仍写入告警记录（ruleName 标记为"未知规则"）
  │   │
  │   ├─ 写入告警记录（monitor_alert_record）
  │   │   字段: ruleId, alertName, severity, instance, labels(JSON),
  │   │         annotations(JSON), status=FIRING, firedAt, tenantId
  │   │
  │   ├─ 检查是否在静默期内（查询 monitor_alert_silence）
  │   │   → 命中静默规则 → 跳过通知，仅记录；告警记录 notified=false
  │   │
  │   ├─ 触发 WebSocket 广播
  │   │   → AlertPushService.broadcast(AlertEventDTO)
  │   │   → 向所有在线 WebSocket Session 推送告警事件 JSON
  │   │
  │   └─ 异步发送多渠道通知（@Async）
  │       → 查询规则绑定的 notifyChannelIds
  │       → 逐渠道调用对应 NotifySender.send()
  │       → 记录通知发送结果（monitor_notify_log）
  │
  ├─[status = resolved]
  │   ├─ 查找对应的 FIRING 记录（by alertName + instance + firedAt）
  │   │   → 找到 → 更新 status=RESOLVED, resolvedAt=endsAt
  │   │   → 未找到 → 记录 warn 日志，忽略
  │   │
  │   └─ 可选：发送告警恢复通知（由规则配置控制）
  │
  └─ 返回 200 OK（AlertManager 需要 200 才不会重试）
```

### 3.8 WebSocket 实时推送流程

**触发条件：** 前端页面建立 WebSocket 连接；告警事件产生时服务端主动推送。

```
前端建立连接：WebSocket ws://{host}/ws/alerts?token={accessToken}
  │
  ├─ WebSocket 握手时校验 Token
  │   → Token 无效 → 拒绝连接（HTTP 401）
  │
  ├─ 连接建立成功
  │   → 注册 Session 至 WebSocketSessionRegistry（Map<userId, session>）
  │   → 发送 CONNECTED 握手确认消息
  │
  ├─ 心跳保活
  │   → 服务端每 30s 向所有连接发送 PING 帧
  │   → 客户端需在 60s 内回复 PONG，否则关闭连接
  │
  ├─[告警事件到达，AlertPushService.broadcast()]
  │   → 遍历 WebSocketSessionRegistry 所有活跃 Session
  │   → 向每个 Session 发送告警事件 JSON
  │       {
  │         type: "ALERT_FIRING",
  │         alertName, severity, instance, summary,
  │         firedAt, recordId
  │       }
  │
  └─[连接断开]
      → 从 WebSocketSessionRegistry 移除 Session
```

### 3.9 通知渠道管理流程

#### 3.9.1 新增通知渠道

```
管理员请求 POST /api/monitor/notify/channels
  │
  ├─ 校验权限（需具备 monitor:notify:channel:add）
  ├─ 校验渠道类型（EMAIL / DINGTALK / WECHAT_WORK / WEBHOOK）
  ├─ 校验渠道配置完整性（不同类型必填字段不同）
  │   EMAIL: smtpHost, smtpPort, senderAddr, receiverAddrs（逗号分隔）
  │   DINGTALK: webhookUrl, secret（加签密钥）
  │   WECHAT_WORK: webhookUrl
  │   WEBHOOK: url, method（GET/POST）, headers（JSON），body 模板
  ├─ 加密存储敏感字段（secret, password）→ AES-256-GCM 加密
  ├─ 写入 monitor_notify_channel
  └─ 返回创建成功的渠道信息（敏感字段脱敏返回）
```

#### 3.9.2 查询/编辑/删除通知渠道

```
GET  /api/monitor/notify/channels         → 分页查询，支持按 channelName / type 过滤
GET  /api/monitor/notify/channels/{id}    → 渠道详情（敏感字段脱敏）
PUT  /api/monitor/notify/channels/{id}    → 编辑渠道配置（重新加密存储敏感字段）
DELETE /api/monitor/notify/channels/{id}  → 校验无规则引用后删除
```

#### 3.9.3 测试通知渠道

```
管理员请求 POST /api/monitor/notify/channels/{id}/test
  │
  ├─ 读取渠道配置（解密敏感字段）
  ├─ 构建测试消息
  │   发送内容："[OpenCloud] 这是一条测试通知消息，渠道配置验证成功"
  ├─ 调用对应 NotifySender.send()
  │   → 成功 → 返回 { success: true, message: "发送成功" }
  │   → 失败 → 返回 { success: false, message: "发送失败: {原始错误}" }
  └─ 记录测试发送日志
```

### 3.10 告警静默管理流程

**触发条件：** 管理员计划维护操作，需要在维护窗口期内屏蔽特定告警通知。

```
新增静默：POST /api/monitor/alerts/silences
  │
  ├─ 校验权限（需具备 monitor:alert:silence:add）
  ├─ 校验参数
  │   → silenceName 必填
  │   → startAt / endAt 必填，endAt > startAt
  │   → matchLabels：Key-Value 标签匹配列表（至少一条），用于匹配告警
  │     如：[{key: "alertname", value: "NodeDown"}, {key: "severity", value: "CRITICAL"}]
  ├─ 写入 monitor_alert_silence（status = ACTIVE）
  ├─ 同步至 AlertManager Silences API（可选）
  └─ 返回静默规则详情

查询静默：GET /api/monitor/alerts/silences
  → 分页查询，支持按 silenceName / status / 时间范围过滤

停止静默：PUT /api/monitor/alerts/silences/{id}/stop
  → 将 status 更新为 EXPIRED，立即生效（告警不再被屏蔽）

删除静默：DELETE /api/monitor/alerts/silences/{id}
  → 仅允许删除已过期或已停止的静默规则

告警到达时检查静默：
  → 遍历 ACTIVE 状态且 startAt <= now <= endAt 的静默规则
  → 告警 labels 中所有 matchLabels 均匹配 → 命中静默，跳过通知
```

### 3.11 告警确认流程

**触发条件：** 运维人员查看到告警后，手动标记该告警为"已处理"。

```
运维人员请求 PUT /api/monitor/alerts/records/{id}/ack
  │
  ├─ 校验权限（需具备 monitor:alert:record:ack）
  ├─ 校验告警记录存在
  ├─ 校验当前状态为 FIRING 或 ACKNOWLEDGED（已 RESOLVED 的告警不需要手动 ACK）
  ├─ 更新 status = ACKNOWLEDGED
  │   → ackBy = 当前用户 ID
  │   → ackAt = now
  │   → ackNote = 请求体中的备注（可选，最大 500 字）
  └─ 返回更新后的告警记录
```

### 3.12 告警记录查询流程

**触发条件：** 运维/管理员查看当前告警列表或历史告警记录。

```
客户端请求 GET /api/monitor/alerts/records（分页列表）
  │
  ├─ 校验权限（需具备 monitor:alert:record:list）
  ├─ 支持查询参数：
  │   alertName（模糊）/ severity / status（FIRING/ACKNOWLEDGED/RESOLVED）
  │   instance / ruleId / beginTime / endTime / page / size
  ├─ 按 tenantId 过滤
  ├─ 默认按 firedAt DESC 排序
  └─ 返回分页列表

客户端请求 GET /api/monitor/alerts/records/{id}（详情）
  │
  ├─ 校验告警记录存在
  └─ 返回完整告警记录
      {
        id, ruleId, ruleName, alertName, severity,
        instance, labels（JSON 解析后对象）,
        annotations（JSON 解析后对象）,
        status, firedAt, resolvedAt,
        ackBy, ackAt, ackNote,
        notified（是否已发送通知）, notifyLogs（通知发送记录列表）
      }
```

### 3.13 告警统计流程

**触发条件：** 前端仪表盘需要展示告警汇总数据。

```
客户端请求 GET /api/monitor/alerts/stats
  │
  ├─ 校验权限（需具备 monitor:alert:record:list）
  ├─ 支持查询参数：period（today / 7d / 30d，默认 today）
  ├─ 按 tenantId 过滤
  ├─ 聚合 monitor_alert_record 表
  │   → total（总触发次数）
  │   → firing（当前未解决数）
  │   → acknowledged（已确认数）
  │   → resolved（已解决数）
  │   → criticalCount / warningCount / infoCount（按级别分类）
  │   → avgDuration（平均持续时间，仅 RESOLVED 记录参与计算）
  └─ 返回统计数据对象
```

### 3.12 通知发送日志查询流程

**触发条件：** 管理员需要查看某次告警的通知发送结果（是否成功、失败原因）。

```
客户端请求 GET /api/monitor/notify/logs
  │
  ├─ 校验权限（需具备 monitor:notify:log:list）
  ├─ 支持查询参数：alertRecordId / channelId / status（SUCCESS/FAILED）/ 时间范围
  ├─ 按 tenantId 隔离
  └─ 返回分页列表
      [{
        id, alertRecordId, channelId, channelName, channelType,
        status, errorMsg, sentAt, retryCount
      }, ...]
```

## 4. 业务规则

### 4.1 权限规则

| 规则 | 说明 |
|------|------|
| 指标查询权限 | 需具备 `monitor:metrics:query` 权限；超级管理员直接放行 |
| 告警规则管理权限 | 增删改需 `monitor:alert:rule:add/edit/delete`；查询需 `monitor:alert:rule:list` |
| 通知渠道管理权限 | 增删改需 `monitor:notify:channel:add/edit/delete`；查询需 `monitor:notify:channel:list` |
| 租户隔离 | 告警规则、告警记录、通知渠道、告警静默均按 tenantId 隔离；超级管理员可跨租户查询 |

### 4.2 指标查询规则

| 规则 | 说明 |
|------|------|
| 时间范围限制 | 范围查询 start~end 不超过 30 天，防止 Prometheus 返回超大数据集 |
| 超时设置 | WebClient 请求 Prometheus 超时 10 秒，超时返回 503 |
| PromQL 注入防护 | 不对用户 PromQL 做限制（专业用户），但记录所有 PromQL 查询日志 |
| 多集群支持 | 通过配置 `opencloud.prometheus.url` 指定 Prometheus 地址；v1 版本单集群 |

### 4.3 告警规则管理规则

| 规则 | 说明 |
|------|------|
| 规则名唯一性 | 同一租户下 ruleName 不可重复 |
| PromQL 校验 | 新增/编辑时通过 Prometheus /api/v1/query 验证 PromQL 语法有效 |
| 严重级别 | 固定三级：INFO（低）/ WARNING（中）/ CRITICAL（高） |
| 持续时间 | duration 字段（如 "5m"）决定持续多久才触发，避免毛刺；格式：[1-60]m，最小 1m，最大 60m |
| AlertManager 同步 | 规则 CRUD 后触发 AlertManager 配置刷新；失败记录日志，不阻塞主流程 |
| 通知渠道绑定 | 一条规则可绑定多个通知渠道；通知渠道可为空（仅入库，不发送通知） |
| 恢复通知 | 规则配置 `notifyOnResolve = true` 时，告警解除后额外发送"告警已恢复"通知 |

### 4.4 告警事件规则

| 规则 | 说明 |
|------|------|
| 幂等写入 | Webhook 可能重复推送，按 alertName + instance + firedAt 去重 |
| 状态流转 | FIRING → ACKNOWLEDGED（人工确认）→ RESOLVED（告警解除），不可逆向流转 |
| 通知重试 | 通知发送失败后，@Async 方式最多重试 3 次（间隔 30s / 60s / 120s） |
| 告警保留 | 告警记录永久保留，不做逻辑删除（审计需要）；可按时间段归档 |

### 4.5 通知渠道规则

| 规则 | 说明 |
|------|------|
| 敏感字段加密 | 钉钉 secret、邮件密码、Webhook 认证 token 均使用 AES-256-GCM 加密存储 |
| 渠道类型固定 | 支持 EMAIL / DINGTALK / WECHAT_WORK / WEBHOOK 四种类型 |
| 删除保护 | 被告警规则引用的渠道不可直接删除，需先解绑或删除关联规则 |
| 测试消息 | 每次测试发送记录日志（渠道 ID、发送时间、结果），不影响正式告警计数 |

### 4.6 告警静默规则

| 规则 | 说明 |
|------|------|
| 时间范围 | startAt / endAt 必须是未来时间；endAt 必须大于 startAt |
| 标签匹配 | matchLabels 中所有条件均需满足才生效（AND 语义），不支持正则 |
| 自动过期 | endAt 到达后静默规则自动失效（status 不变，查询时按时间过滤） |
| 手动停止 | 可提前手动停止静默（status = EXPIRED），立即生效 |

### 4.7 WebSocket 规则

| 规则 | 说明 |
|------|------|
| 认证必须 | WebSocket 握手时携带 Token（query 参数），无效 Token 拒绝连接 |
| 每用户单连接 | 同一用户多次连接时，新连接替换旧连接（避免资源堆积） |
| 心跳保活 | 服务端每 30s 发送 PING，客户端 60s 未响应自动断开 |
| 推送失败处理 | 某 Session 推送失败（已断开）时，从注册表移除，不影响其他 Session |

### 4.8 通知消息模板规则

| 类型 | 消息内容规范 |
|------|------------|
| EMAIL | 主题：`[告警级别] alertName - instance`；正文：HTML 模板，包含规则名、级别、实例、触发时间、描述、处理建议 |
| DINGTALK | Markdown 格式；标题：`[告警级别] alertName`；正文：触发时间 / 实例 / 描述 / 快捷链接；CRITICAL 级别 @ 所有人 |
| WECHAT_WORK | Markdown 格式（企微机器人）；同钉钉格式 |
| WEBHOOK | 按渠道配置的 bodyTemplate 渲染（Freemarker 模板），可访问：alertName / severity / instance / summary / firedAt / recordId |

---

### 4.9 告警统计规则

| 规则 | 说明 |
|------|------|
| 统计维度 | 按租户、按严重级别、按时间段（今日/近7天/近30天） |
| 统计内容 | 触发总数 / 已确认数 / 已解决数 / 平均持续时间 / 未解决数 |
| 数据来源 | 实时聚合 monitor_alert_record 表，不使用独立统计表（v1） |
| 接口 | `GET /api/monitor/alerts/stats` |

## 5. 业务异常场景

### 5.1 指标查询相关

| 场景 | HTTP 状态 | 错误码 | 处理方式 |

| PromQL 语法错误 | 400 | MONITOR_1002 | 返回 Prometheus 原始错误信息 |

### 5.2 Grafana 集成相关

|------|-----------|--------|---------|

| Dashboard 不存在 | 404 | MONITOR_2002 | 返回"面板不存在或已被删除" |
| Token 无效 | 503 | MONITOR_2003 | 返回"Grafana 认证失败，请检查服务配置" |

### 5.3 告警规则相关

### 5.4 告警记录相关

## 6. 关键数据说明

### 6.1 告警记录状态流转

```
FIRING（触发中）
  │
  ├─[人工确认] → ACKNOWLEDGED（已确认）
  │
  └─[告警解除（AlertManager resolved）] → RESOLVED（已解决）

ACKNOWLEDGED
  └─[告警解除（AlertManager resolved）] → RESOLVED（已解决）
```

### 6.2 告警严重级别

| 级别 | 说明 | 前端颜色建议 |
|------|------|------------|
| INFO | 提示性告警，不影响服务 | 蓝色 |
| WARNING | 预警，需要关注 | 橙色/黄色 |
| CRITICAL | 严重，服务已受影响 | 红色 |

### 6.3 通知渠道类型与配置字段

| 类型 | 必填字段 | 选填字段 |
|------|---------|---------|
| EMAIL | senderAddr, receiverAddrs（逗号分隔），smtpHost, smtpPort, smtpUser, smtpPass | subject 模板, body 模板 |
| DINGTALK | webhookUrl | secret（加签），at（@ 手机号列表） |
| WECHAT_WORK | webhookUrl | mentioned_list（@ 成员列表） |
| WEBHOOK | url | method（默认 POST），headers（JSON），bodyTemplate（Freemarker 模板）|

### 6.4 WebSocket 消息格式

```json
// 告警触发
{
  "type": "ALERT_FIRING",
  "recordId": 123,
  "alertName": "NodeMemoryHigh",
  "severity": "WARNING",
  "instance": "192.168.1.10:9100",
  "summary": "节点内存使用率超过 80%",
  "firedAt": "2026-03-31T10:00:00"
}

// 告警解除
{
  "type": "ALERT_RESOLVED",
  "recordId": 123,
  "alertName": "NodeMemoryHigh",
  "resolvedAt": "2026-03-31T10:30:00"
}

// 心跳
{
  "type": "PING"
}
```
