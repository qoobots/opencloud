# opencloud-system 业务设计文档

---

## 1. 业务背景

opencloud-system 模块承载 OpenCloud 云管平台的系统管理能力，是整个平台数据与权限模型的核心维护中心。auth 模块负责认证鉴权，但其所依赖的用户、角色、菜单、权限数据均由 system 模块统一维护和管理。

### 1.1 使用角色

| 角色 | 说明 |
|------|------|
| 超级管理员 | 平台全局权限，可管理所有租户下的用户、角色、菜单；不受 RBAC 限制 |
| 租户管理员 | 在所属租户范围内管理用户和角色；只能操作同租户数据 |
| 普通用户 | 只能查看和修改自己的个人信息；无管理操作权限 |

### 1.2 核心职责

| 职责 | 说明 |
|------|------|
| 用户管理 | 维护系统用户信息（基础信息、状态、角色绑定、密码重置） |
| 角色管理 | 维护角色定义与角色-菜单/权限绑定关系 |
| 菜单权限管理 | 维护菜单树（目录/菜单/按钮），定义权限标识（permission code） |
| 部门管理 | 维护部门/组织架构树，用户可归属某一部门 |
| 字典管理 | 维护平台各类枚举字典（如用户状态、操作类型等） |
| 操作日志审计 | 通过 AOP 切面自动记录管理操作日志，支持查询和导出 |
| 登录日志查询 | 向管理员暴露 auth 模块写入的登录日志，支持分页查询 |
| 权限缓存联动 | 角色/权限/用户状态变更时，主动清除 auth 模块中对应的权限缓存 |

---

## 2. 业务目标

1. 提供完整的用户-角色-菜单-权限（RBAC）数据维护入口
2. 支持多租户数据隔离，不同租户的用户、角色、菜单互不干扰
3. 提供部门/组织架构管理，支持用户按部门归属
4. 提供字典数据维护，规范化枚举类型数据
5. 提供完整的操作日志和登录日志审计查询能力
6. 角色/权限变更时主动通知 auth 模块刷新权限缓存，保证权限实时生效

## 3. 核心业务流程

### 3.1 用户管理流程

#### 3.1.1 创建用户

**触发条件：** 租户管理员在用户管理页点击「新增用户」并提交表单。

```
管理员提交新增用户请求
  │
  ├─ 校验操作者权限（需具备 system:user:add 权限）
  │
  ├─ 校验参数
  │   → username 格式（字母/数字/下划线，4~32 位）
  │   → username 在当前租户内唯一（查 sys_user）
  │   → email 格式合法（若填写）
  │   → 手机号格式合法（若填写）
  │
  ├─ 创建 sys_user 记录
  │   → 分配 user_id（UUID）
  │   → 状态默认 ACTIVE
  │   → tenant_id 取操作者的 tenantId
  │
  ├─ 创建 auth_user_credential 记录
  │   → 初始密码：若管理员指定则使用，否则使用系统默认密码
  │   → BCrypt 加密存储
  │
  ├─ 绑定角色（若指定）
  │   → 批量插入 sys_user_role 关联记录
  │
  ├─ 写入操作日志（sys_operation_log）
  │   → operationType=CREATE_USER
  │
  └─ 返回创建成功的用户信息
```

#### 3.1.2 编辑用户

**触发条件：** 管理员修改用户基本信息（昵称、邮箱、手机、头像、部门等）。

```
管理员提交编辑用户请求（PUT /api/system/users/{id}）
  │
  ├─ 校验操作者权限（system:user:edit）
  ├─ 校验目标用户存在且属于同一租户
  │
  ├─ 不允许修改 username（作为唯一标识不可变）
  │
  ├─ 更新 sys_user 字段（nickname / email / phone / avatar / dept_id 等）
  │
  ├─ 写入操作日志
  │
  └─ 返回 200 OK
```

#### 3.1.3 删除用户

**触发条件：** 管理员删除指定用户。

```
管理员请求 DELETE /api/system/users/{id}
  │
  ├─ 校验操作者权限（system:user:delete）
  ├─ 目标用户属于同一租户
  ├─ 不允许删除 admin 超级管理员账号
  ├─ 不允许删除自己
  │
  ├─ 逻辑删除 sys_user（deleted=1）
  ├─ 同步逻辑删除 auth_user_credential（deleted_at=now）
  ├─ 删除 sys_user_role 关联记录
  │
  ├─ 清除 auth 权限缓存（DEL auth:perm:{userId}）
  │
  ├─ 写入操作日志
  │
  └─ 返回 200 OK
```

#### 3.1.4 启用 / 禁用用户

**触发条件：** 管理员切换用户状态。

```
管理员请求 PUT /api/system/users/{id}/status
请求体: { status: "ACTIVE" | "DISABLED" }
  │
  ├─ 校验操作者权限（system:user:edit）
  ├─ 不允许禁用 admin 账号
  ├─ 不允许操作自身账号
  │
  ├─ 更新 sys_user.status
  ├─ 同步更新 auth_user_credential.status
  │
  ├─[禁用用户]
  │   → 清除权限缓存（DEL auth:perm:{userId}）
  │   → 踢出登录（INCR auth:token:version:{userId}，DEL auth:refresh:{userId}:*）
  │
  ├─ 写入操作日志
  │
  └─ 返回 200 OK
```

#### 3.1.5 重置密码

**触发条件：** 管理员为用户重置密码（不需要旧密码）。

```
管理员请求 PUT /api/system/users/{id}/password
请求体: { newPassword }（可选，否则使用系统默认密码）
  │
  ├─ 校验操作者权限（system:user:resetPassword）
  ├─ 校验密码强度（同 auth 模块规则）
  │
  ├─ BCrypt 加密新密码，更新 auth_user_credential.password_hash
  ├─ 更新 password_update_time = now
  │
  ├─ 清除该用户所有 RefreshToken（DEL auth:refresh:{userId}:*）
  ├─ INCR auth:token:version:{userId}（强制重新登录）
  │
  ├─ 写入操作日志（operationType=RESET_PASSWORD）
  │
  └─ 返回 200 OK
```

#### 3.1.6 分配角色

**触发条件：** 管理员为用户批量设置角色。

```
管理员请求 PUT /api/system/users/{id}/roles
请求体: { roleIds: [1, 2, 3] }
  │
  ├─ 校验操作者权限（system:user:assignRole）
  ├─ 角色必须属于同一租户
  │
  ├─ 先删除 sys_user_role 中该用户的所有记录
  ├─ 再批量插入新角色绑定记录
  │
  ├─ 清除权限缓存（DEL auth:perm:{userId}）
  │
  ├─ 写入操作日志（operationType=ASSIGN_ROLE）
  │
  └─ 返回 200 OK
```

#### 3.1.7 查询用户列表 / 详情

| 操作 | 说明 |
|------|------|
| 分页列表 | 支持按用户名、昵称、状态、部门、创建时间范围筛选 |
| 用户详情 | 返回用户基础信息 + 已绑定角色列表 |
| 下拉选项 | 精简字段（id + username + nickname），用于角色分配等场景的选人 |

### 3.2 角色管理流程

#### 3.2.1 创建角色

```
管理员请求 POST /api/system/roles
请求体: { roleName, roleCode, description, sort, status }
  │
  ├─ 校验操作者权限（system:role:add）
  ├─ roleCode 在当前租户内唯一
  ├─ roleCode 格式规范（大写字母/数字/下划线）
  │
  ├─ 插入 sys_role 记录
  │   → tenant_id 取操作者 tenantId
  │   → status 默认 ACTIVE
  │
  ├─ 写入操作日志
  │
  └─ 返回创建的角色信息
```

#### 3.2.2 编辑角色

```
管理员请求 PUT /api/system/roles/{id}
  │
  ├─ 校验操作者权限（system:role:edit）
  ├─ 内置角色（is_builtin=true）不允许修改 roleCode
  ├─ 修改 roleName / description / sort / status
  │
  ├─[禁用角色]
  │   → 查询该角色下所有用户（sys_user_role）
  │   → 批量清除这些用户的权限缓存（DEL auth:perm:{userId}）
  │
  ├─ 写入操作日志
  │
  └─ 返回 200 OK
```

#### 3.2.3 删除角色

```
管理员请求 DELETE /api/system/roles/{id}
  │
  ├─ 校验操作者权限（system:role:delete）
  ├─ 内置角色（is_builtin=true）不允许删除
  ├─ 检查是否有用户绑定该角色
  │   → 有用户绑定 → 返回 400（角色使用中，无法删除）
  │
  ├─ 删除 sys_role_menu 关联记录
  ├─ 逻辑删除 sys_role
  │
  ├─ 写入操作日志
  │
  └─ 返回 200 OK
```

#### 3.2.4 分配菜单权限

**触发条件：** 管理员为角色配置可访问的菜单和按钮权限。

```
管理员请求 PUT /api/system/roles/{id}/menus
请求体: { menuIds: [1, 2, 5, 10, 11] }
  │
  ├─ 校验操作者权限（system:role:assignMenu）
  ├─ 内置超级管理员角色不允许修改权限（权限全开）
  │
  ├─ 先删除 sys_role_menu 中该角色的所有记录
  ├─ 再批量插入新菜单/权限关联
  │
  ├─ 查询该角色下所有用户（sys_user_role）
  ├─ 批量清除这些用户的权限缓存（DEL auth:perm:{userId}）
  │
  ├─ 写入操作日志（operationType=ASSIGN_MENU）
  │
  └─ 返回 200 OK
```

### 3.3 菜单权限管理流程

#### 3.3.1 菜单结构说明

菜单分为三种类型，使用 `menu_type` 区分：

| 类型值 | 类型名 | 说明 |
|--------|--------|------|
| 0 | 目录 | 侧边栏父节点，不对应具体页面，无路由 |
| 1 | 菜单 | 对应前端路由页面，有 path / component |
| 2 | 按钮/权限 | 页面内操作按钮，不参与路由，仅携带 permission 标识 |

**权限标识（permission code）** 格式约定：`模块:资源:操作`，例如：
- `system:user:list` — 用户列表查看
- `system:user:add` — 新增用户
- `system:role:assignMenu` — 角色分配菜单

#### 3.3.2 菜单的增删改查

```
新增菜单（POST /api/system/menus）
  │
  ├─ 校验操作者权限（system:menu:add）
  ├─ 父节点存在（parent_id 不为 0 时校验）
  ├─ 同父节点下 menu_name 不重复
  ├─ 按钮类型不能有子节点
  │
  ├─ 插入 sys_menu 记录
  ├─ 写入操作日志
  └─ 返回新增菜单信息

编辑菜单（PUT /api/system/menus/{id}）
  │
  ├─ 不允许将自身设为子孙节点的父节点（防止循环引用）
  ├─ 修改 menu_type 时，若有子节点则拒绝
  └─ 修改 permission 字段后，批量清除关联角色的所有用户权限缓存

删除菜单（DELETE /api/system/menus/{id}）
  │
  ├─ 检查是否有子节点，有则拒绝删除（需先清空子节点）
  ├─ 检查是否有角色绑定此菜单（sys_role_menu），有则拒绝
  ├─ 逻辑删除 sys_menu
  └─ 清除所有关联角色用户的权限缓存
```

#### 3.3.3 当前用户菜单树

```
GET /api/system/menus/current
  │
  ├─ 从 SecurityContext 取 userId + roles
  │
  ├─[SUPER_ADMIN]→ 返回全量菜单树（含所有按钮权限）
  │
  ├─[普通角色]→ 查询该用户所有角色 → 查 sys_role_menu → 获取菜单 ID 集合
  │             → 查 sys_menu（仅 menu_type=0/1 的目录和菜单节点，不含按钮）
  │             → 递归构建树形结构
  │
  └─ 返回菜单树（供前端渲染侧边栏）
```

### 3.4 部门管理流程

#### 3.4.1 部门结构说明

部门采用树形结构，支持多级嵌套（建议不超过 5 层）。每个租户维护独立的部门树，根节点 `parent_id = 0`。

#### 3.4.2 部门的增删改查

```
新增部门（POST /api/system/depts）
  │
  ├─ 校验操作者权限（system:dept:add）
  ├─ 父节点存在且属于同一租户
  ├─ 同父节点下 dept_name 不重复
  ├─ 层级深度不超过 5 层
  │
  ├─ 插入 sys_dept 记录（tenant_id 取操作者 tenantId）
  └─ 返回新增部门信息

编辑部门（PUT /api/system/depts/{id}）
  │
  ├─ 不允许将自身设为子孙节点的父节点
  └─ 更新 dept_name / sort / leader / phone / email 等字段

删除部门（DELETE /api/system/depts/{id}）
  │
  ├─ 检查是否有子部门 → 有则拒绝
  ├─ 检查是否有用户归属此部门 → 有则拒绝
  └─ 逻辑删除 sys_dept
```

### 3.5 字典管理流程

#### 3.5.1 字典结构说明

字典分两层：
- **字典类型（sys_dict_type）**：如 `sys_user_status`（用户状态）、`sys_yes_no`（是否）
- **字典数据（sys_dict_data）**：隶属于某类型的具体键值对，如 `ACTIVE=正常`、`DISABLED=禁用`

#### 3.5.2 字典的增删改查

```
字典类型管理：
  新增 POST /api/system/dict/types    → 校验 dict_type 唯一
  编辑 PUT  /api/system/dict/types/{id}
  删除 DELETE /api/system/dict/types/{id} → 检查是否有字典数据，有则连带删除

字典数据管理：
  新增 POST /api/system/dict/data    → 校验同类型下 dict_value 唯一
  编辑 PUT  /api/system/dict/data/{id}
  删除 DELETE /api/system/dict/data/{id}
  按类型查询 GET /api/system/dict/data?type=xxx → 供前端下拉列表使用
```

### 3.6 操作日志流程

#### 3.6.1 日志自动记录（AOP）

**触发条件：** 标注了 `@SysLog` 注解的 Controller 方法被调用时，AOP 切面自动拦截并异步记录操作日志。

```
请求到达 Controller
  │
  ├─ AOP 前置：记录请求时间、操作人（从 SecurityContext 取）、IP、接口路径、HTTP 方法、请求参数
  │
  ├─ 执行业务逻辑
  │
  ├─ AOP 后置：记录响应结果（success/fail）、响应耗时
  │
  ├─ 异步写入 sys_operation_log（@Async，不阻塞主流程）
  │   字段: userId, tenantId, username, moduleName, operationType, method,
  │         requestUrl, requestParam, responseResult, status, costTime, operateIp, operateTime
  │
  └─ 异常情况：记录异常信息（exceptionMsg）、status=FAILED
```

#### 3.6.2 日志查询

| 操作 | 说明 |
|------|------|
| 分页列表 | 按用户名/模块/操作类型/状态/时间范围筛选 |
| 日志详情 | 查看完整请求参数和响应结果 |
| 日志导出 | 导出 Excel（EasyExcel），按当前筛选条件导出 |

### 3.7 登录日志查询流程

> 登录日志由 auth 模块负责写入（`auth_login_log` 表），system 模块只提供查询视图，不负责写入。

```
管理员 GET /api/system/logs/login
  │
  ├─ 校验操作者权限（system:log:login）
  │
  ├─ 筛选条件：
  │   → username（模糊匹配）
  │   → login_ip（精确匹配）
  │   → status（SUCCESS / FAILED / LOGOUT / FORCE_LOGOUT）
  │   → 登录时间范围（loginTimeStart / loginTimeEnd）
  │
  ├─ 分页查询 auth.auth_login_log（跨 Schema 访问，同一 DB）
  │
  ├─ 返回字段：
  │   id, username, login_ip, user_agent, login_time, logout_time,
  │   status, fail_reason, created_at
  │
  └─ 支持导出 Excel（GET /api/system/logs/login/export，max 10000 条）
```

### 3.8 个人中心流程

**触发条件：** 任意已登录用户查看或修改自己的个人信息。

```
获取个人信息（GET /api/system/profile）
  │
  ├─ 从 SecurityContext 取 userId
  ├─ 查询 sys_user（基础信息）
  ├─ 查询绑定角色列表
  └─ 返回个人信息（不含密码字段）

修改个人信息（PUT /api/system/profile）
请求体: { nickname, email, phone, avatar }
  │
  ├─ 校验 email / phone 格式
  ├─ 更新 sys_user 对应字段
  └─ 返回更新后的个人信息
```

> **说明：** 修改密码由 auth 模块负责（`POST /auth/password/change`），个人中心不重复提供。

### 3.9 租户管理流程

> 租户管理仅超级管理员可见，租户管理员无权访问。

#### 3.9.1 租户的增删改查

```
新增租户（POST /api/system/tenants）
  │
  ├─ 校验操作者为超级管理员（role=SUPER_ADMIN）
  ├─ tenant_code 全局唯一
  ├─ 插入 sys_tenant 记录（status=ACTIVE，expire_time 可选）
  │
  ├─ 自动创建租户默认管理员账号（username=admin@{tenantCode}）
  │   → 同时写入 sys_user + auth_user_credential
  └─ 返回租户信息

编辑租户（PUT /api/system/tenants/{id}）
  │
  ├─ 允许修改 tenant_name / contact_name / contact_email / expire_time
  └─ 不允许修改 tenant_code

删除租户（DELETE /api/system/tenants/{id}）
  │
  ├─ 不允许删除默认租户（is_default=true）
  ├─ 检查租户下是否存在用户 → 有则拒绝
  └─ 逻辑删除 sys_tenant

启用/禁用租户（PUT /api/system/tenants/{id}/status）
  │
  ├─[禁用]→ 批量清除该租户所有用户的权限缓存（auth:perm:{userId}）
  │         → 批量 INCR auth:token:version:{userId}（强制所有用户下线）
  └─ 更新 sys_tenant.status
```

#### 3.9.2 查询租户列表

| 操作 | 说明 |
|------|------|
| 分页列表 | 支持按租户名、状态、创建时间范围筛选 |
| 租户详情 | 返回租户信息 + 当前用户数量 |
| 下拉选项 | 精简字段（id + tenant_name），供超级管理员切换视角时使用 |

### 3.10 权限缓存联动

当系统数据发生以下变更时，必须主动清除 auth 模块的 Redis 权限缓存，否则权限变更最长延迟 10 分钟才生效：

| 触发操作 | 影响范围 | 需清除的缓存 |
|---------|---------|------------|
| 修改用户角色绑定（`PUT /api/system/users/{id}/roles`） | 单个用户 | `DEL auth:perm:{userId}` |
| 禁用 / 删除用户 | 单个用户 | `DEL auth:perm:{userId}` |
| 修改角色状态为禁用（`PUT /api/system/roles/{id}`） | 角色下所有用户 | 批量 `DEL auth:perm:{userId}` |
| 修改角色菜单权限（`PUT /api/system/roles/{id}/menus`） | 角色下所有用户 | 批量 `DEL auth:perm:{userId}` |
| 删除角色 | 角色下所有用户 | 批量 `DEL auth:perm:{userId}` |
| 修改菜单 permission 字段 | 关联该菜单的所有角色下所有用户 | 批量 `DEL auth:perm:{userId}` |
| 删除菜单 | 关联该菜单的所有角色下所有用户 | 批量 `DEL auth:perm:{userId}` |
| 禁用租户 | 租户下所有用户 | 批量 `DEL auth:perm:{userId}` + `INCR auth:token:version:{userId}` |

**实现方式：** system 模块直接操作 Redis（共享同一 Redis 实例，命名空间 `auth:`），在业务 Service 的事务提交后（Spring 事务事件 `@TransactionalEventListener(phase=AFTER_COMMIT)`）执行缓存清除，避免事务回滚后缓存被误清除。

**批量清除用户范围查询：**
```sql
-- 角色下所有用户
SELECT user_id FROM system.sys_user_role WHERE role_id = ?

-- 菜单关联角色下所有用户
SELECT DISTINCT ur.user_id
FROM system.sys_role_menu rm
JOIN system.sys_user_role ur ON rm.role_id = ur.role_id
WHERE rm.menu_id = ?

-- 租户下所有用户
SELECT id FROM system.sys_user WHERE tenant_id = ? AND deleted = 0
```

## 4. 业务规则

### 4.1 用户管理规则

| 规则 | 说明 |
|------|------|
| 用户名格式 | 4~32 位，只允许字母、数字、下划线；字母开头 |
| 用户名唯一性 | 同一租户内 username 唯一；不同租户允许重名 |
| 邮箱格式 | 符合标准 email 格式（选填） |
| 手机号格式 | 中国大陆 11 位手机号（选填） |
| 超级管理员保护 | admin 账号不允许删除、禁用、修改用户名 |
| 自操作限制 | 管理员不允许删除自身、禁用自身账号 |
| 初始密码规则 | 创建用户时可由管理员指定密码，或使用系统默认密码 `Welcome@2026`；密码须满足强度要求（≥8位，含大小写+数字） |
| 软删除 | sys_user 使用 `deleted=1` 逻辑删除；auth_user_credential 使用 `deleted_at` 软删除 |
| 租户隔离 | 所有用户操作均隐式附带 tenant_id 过滤；租户管理员不能访问其他租户的用户数据 |

### 4.2 角色管理规则

| 规则 | 说明 |
|------|------|
| 角色 Code 唯一性 | 同一租户内 role_code 唯一 |
| 内置角色保护 | `is_builtin=true` 的角色（如 SUPER_ADMIN）不允许删除、不允许修改 role_code |
| 角色使用中不可删除 | 有用户绑定的角色不允许直接删除，需先解绑用户 |
| 超级管理员权限全开 | SUPER_ADMIN 角色不绑定具体菜单，由 auth 模块跳过权限校验直接放行 |

### 4.3 菜单管理规则

| 规则 | 说明 |
|------|------|
| 层级约束 | 目录 → 菜单 → 按钮，不允许跨层绑定（如按钮下不能有子节点） |
| 按钮不参与路由 | menu_type=2 的节点只携带 permission 标识，前端不渲染为菜单项 |
| permission 全局唯一 | permission code 在全系统唯一，与后端 `@PreAuthorize` 注解对应 |
| 删除有子节点拦截 | 父节点存在子节点时不允许删除 |
| 删除有角色绑定拦截 | 有角色绑定此菜单时不允许删除 |

### 4.4 部门管理规则

| 规则 | 说明 |
|------|------|
| 层级深度限制 | 最多 5 层 |
| 有子部门不可删除 | 需先删除所有子部门 |
| 有用户归属不可删除 | 需先将用户迁移到其他部门 |
| 租户隔离 | 部门树与租户绑定，跨租户不可见 |

### 4.5 字典管理规则

| 规则 | 说明 |
|------|------|
| 字典类型 Code 唯一 | dict_type 字段全局唯一 |
| 字典数据 Value 唯一 | 同一字典类型下 dict_value 唯一 |
| 系统内置字典保护 | is_system=true 的字典类型不允许删除 |

### 4.6 操作日志规则

| 规则 | 说明 |
|------|------|
| 触发方式 | Controller 方法标注 `@SysLog(module="用户管理", action="新增用户")` 注解，AOP 切面自动拦截 |
| 异步写入 | `@Async` 方式写入，不影响业务主流程性能；写入失败不影响业务响应 |
| 敏感字段脱敏 | 请求体/响应体中的 `password`、`passwordHash`、`token` 字段在序列化为日志前替换为 `***` |
| 只读归档 | 日志只追加写入，不提供修改和删除接口（即使是超级管理员也不允许通过界面删除） |
| 数据保留 | 操作日志保留 180 天，定时任务（每日凌晨 2:00）清理过期记录 |
| 分页限制 | 单次查询最大 500 条，导出最大 10000 条 |
| 记录范围 | 所有 POST / PUT / DELETE 操作均记录；GET 查询不记录（除导出操作外） |
